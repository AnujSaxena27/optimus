# Configuration Guide\n\n## Environment Variables\n\n### Required Variables\n\n#### Groq API\n```\nGROQ_API_KEY=your_groq_api_key_here\n```\nGet your API key from [Groq Console](https://console.groq.com)\n\n#### Supabase Auth\n```\nNEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co\nNEXT_PUBLIC_SUPABASE_ANON_KEY=your_supabase_anon_key_here\n```\n\n### Optional Configuration\n\n#### Application\n```\nNODE_ENV=development|production  # Default: development\nLOG_LEVEL=DEBUG|INFO|WARN|ERROR  # Default: INFO\nPORT=3000                         # Default: 3000\n```\n\n#### Queue and Batching (Tuning)\n```\nBATCH_SIZE=5                      # Requests per batch (default: 5)\nBATCH_TIMEOUT_MS=100              # Max wait time (default: 100ms)\nMAX_QUEUE_SIZE=1000               # Max queue size (default: 1000)\nREQUEST_TIMEOUT_MS=30000          # Request timeout (default: 30s)\n```\n\n#### Caching (Tuning)\n```\nCACHE_TTL_SECONDS=3600            # Cache TTL (default: 1 hour)\nCACHE_MAX_ENTRIES=10000           # Max cache size (default: 10,000)\nCACHE_ENABLE_COMPRESSION=true     # Compress cache (default: true)\n```\n\n#### Rate Limiting (Tuning)\n```\nRATE_LIMIT_WINDOW_MS=60000        # Rate limit window (default: 1 minute)\nRATE_LIMIT_MAX_REQUESTS=100       # Requests per window (default: 100)\n```\n\n## Constants Configuration\n\nEdit `src/config/constants.ts` to adjust system behavior:\n\n### Batch Processing Tuning\n\n```typescript\nBATCH_SIZE: 5          // Increase for higher throughput, decrease for lower latency\nBATCH_TIMEOUT_MS: 100  // Shorter = lower latency but smaller batches\nMAX_QUEUE_SIZE: 1000   // Increase if you have high volume\n```\n\n**Recommendation**:\n- **Low Latency**: BATCH_SIZE=1, BATCH_TIMEOUT_MS=10\n- **Balanced**: BATCH_SIZE=5, BATCH_TIMEOUT_MS=100\n- **High Throughput**: BATCH_SIZE=20, BATCH_TIMEOUT_MS=500\n\n### Cache Tuning\n\n```typescript\nCACHE_TTL_SECONDS: 3600       // 1 hour cache\nCACHE_MAX_ENTRIES: 10000      // Maximum cache entries\n```\n\n**Recommendation**:\n- **Real-time Apps**: CACHE_TTL_SECONDS=60 (1 minute)\n- **Standard**: CACHE_TTL_SECONDS=3600 (1 hour)\n- **Historical Data**: CACHE_TTL_SECONDS=86400 (1 day)\n\n### Rate Limiting Tuning\n\n```typescript\nRATE_LIMIT_WINDOW_MS: 60000    // 1 minute window\nRATE_LIMIT_MAX_REQUESTS: 100   // 100 requests per minute\n```\n\n**Recommendation**:\n- **Free Tier**: 100 req/min\n- **Professional**: 1000 req/min\n- **Enterprise**: 10000 req/min (or integrate Redis)\n\n### Model Configuration\n\nInside `MODEL_CONFIGS`:\n\n```typescript\n'model-id': {\n  maxTokens: 32768,        // Model context window\n  costPerMTok: 0.24,       // Cost per million tokens\n  latencyMs: 150,          // Typical latency\n  provider: 'Groq',        // Provider name\n}\n```\n\n## Performance Tuning\n\n### For Lower Latency\n\n1. Reduce batch timeout\n2. Increase cache TTL\n3. Lower batch size\n4. Enable connection pooling (automatic)\n\n```typescript\nBATCH_TIMEOUT_MS: 10,\nCACHE_TTL_SECONDS: 7200,\nBATCH_SIZE: 1,\n```\n\n### For Higher Throughput\n\n1. Increase batch size\n2. Increase batch timeout\n3. Increase max queue size\n4. Configure Redis for distributed caching\n\n```typescript\nBATCH_SIZE: 20,\nBATCH_TIMEOUT_MS: 500,\nMAX_QUEUE_SIZE: 5000,\n```\n\n### For Cost Optimization\n\n1. Increase cache TTL (reuse responses)\n2. Route to cheaper models when possible\n3. Batch similar requests\n4. Monitor cost per request\n\n## Scaling Considerations\n\n### Vertical Scaling (Single Server)\n\n- Increase `CACHE_MAX_ENTRIES` (more RAM)\n- Increase `MAX_QUEUE_SIZE` (more RAM)\n- Adjust batch size based on CPU\n\n### Horizontal Scaling (Multiple Servers)\n\n1. **Replace In-Memory Cache with Redis**\n   ```typescript\n   // Use redis service instead of memory cache\n   import Redis from 'redis';\n   ```\n\n2. **Use Distributed Queuing**\n   ```typescript\n   // Use Bull/RabbitMQ instead of in-memory queue\n   import BullQueue from 'bull';\n   ```\n\n3. **Shared Database for Logs**\n   - Already configured with Supabase\n   - Logs from all servers write to same database\n\n### Docker Deployment\n\nSee [DEPLOYMENT.md](./DEPLOYMENT.md) for Docker configuration.\n\n## Monitoring Configuration\n\n### Logs\n\n- Set `LOG_LEVEL=DEBUG` for development\n- Set `LOG_LEVEL=INFO` for production\n- Set `LOG_LEVEL=ERROR` for high-volume production\n\n### Metrics to Monitor\n\n1. **Cache Hit Rate**\n   - Target: >50% for typical applications\n   - Lower indicates low hit rate, increase TTL\n\n2. **Average Response Time**\n   - Target: <500ms for most use cases\n   - Increase batch size if limited by queue\n\n3. **Queue Size**\n   - Monitor: Should stay below MAX_QUEUE_SIZE\n   - Action: Scale horizontally if constantly high\n\n4. **Error Rate**\n   - Target: <1%\n   - Investigate spikes immediately\n\n5. **Cost per Request**\n   - Monitor trends\n   - Optimize model selection\n\n## Troubleshooting\n\n### High Latency\n1. Check cache hit rate (enable cache)\n2. Reduce batch timeout\n3. Check Groq API status\n4. Monitor CPU/Memory usage\n\n### High Error Rate\n1. Check Groq API key\n2. Review error logs\n3. Check rate limits\n4. Verify JWT tokens\n\n### Memory Issues\n1. Reduce CACHE_MAX_ENTRIES\n2. Reduce MAX_QUEUE_SIZE\n3. Reduce BATCH_SIZE\n4. Monitor cache entries\n\n### Request Timeout\n1. Increase REQUEST_TIMEOUT_MS\n2. Check network connectivity\n3. Check Groq API status\n4. Verify message size\n
